## プロダクト全体像

### 30秒サマリ

> 自然言語の定性的な制約を、最適化ソルバーが食える数式(type付きJSON → ハンドラ → ソルバーAPI)に変換する**汎用エージェント**。
> 新しい要件が来たらAIが自分で処理ロジック(ハンドラ)を書き、テストし、人間に承認を求める。

### コア価値(ハッカソン応募観点)

「AIが新しい要件を検出 → 自分で処理ロジックを書く → テストする → 動いたら人間に承認を求める」という**自律エージェントの振る舞い**が成立する。

### デモドメイン

**飲食店シフト**(30人 × 1ヶ月)

- ハッカソン用の**独立したアプリ**として開発（外部サービス連携は見送り）

## システム構成

### 6つの構成要素

| # | 要素 | 役割 |
|---|---|---|
| 1 | Geminiパーサ | 自然言語 → `{type, params}` JSON 変換 |
| 2 | type辞書 | 各typeのJSONスキーマ定義 |
| 3 | マスタ | カテゴリ値の正規化辞書 |
| 4 | ハンドラ辞書+関数 | typeごとの「JSON → ソルバーAPI翻訳係」 |
| 5 | ソルバー | OR-Tools(CP-SAT) |
| 6 | AIエージェント(管理人) | 未知type検出 / ハンドラ生成 / テスト / 承認依頼 |

### 自動化レベル

- **L1**: JSONスキーマ拡張提案 ✅ 含む
- **L2**: ハンドラ関数の自動生成(コード生成・テスト・承認) ✅ **コア**
- L3: ソフト制約の重み自動調整 ⏸ 余裕あれば
- L4以降: 対象外

## アーキテクチャ図

```
[ブラウザ]
   ↓
[Streamlit on Cloud Run]    ← デモUI(承認画面など)
   ↓ APIコール
[FastAPI on Cloud Run]      ← エージェント本体
   ├─ Gemini API (Flash)    ← NL→JSON、分類、正規化
   ├─ Gemini API (Pro)      ← ハンドラ生成、自信度評価
   ├─ OR-Tools              ← ソルバー実行
   └─ Sandbox (subprocess)  ← 生成コードのテスト実行
   ↓
[Supabase (PostgreSQL + JSONB)]  ← 全データ永続化
```

## 重要な設計判断ログ

| 判断 | 理由 |
|---|---|
| ドメインを飲食店シフトに | 30人規模感のリアル感、共感度高い |
| ソルバーは OR-Tools | 公式ドキュメント充実、Pythonサポート最強 |
| DBは Supabase | JSONBで柔軟性確保、管理画面が便利 |
| サンドボックスはsubprocess | ハッカソンのリスクモデルに十分、実装爆速 |
| Gemini Flash + Proカスケード | コスト効率、自信度ベースでProフォールバック |
| 入力はCSV + 自然言語のハイブリッド | 実用性とAI必然性の両立 |
| 出勤希望ベース(休み希望ではない) | 飲食店バイト文化に適合 |
| 30分ブロック単位 | 飲食店のピーク時間刻みを表現 |
