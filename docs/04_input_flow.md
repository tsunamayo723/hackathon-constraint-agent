## 入力経路の全体像

シフト作成における制約情報は3経路で入る:

| 経路 | 内容 | 入力方法 |
|---|---|---|
| **A: 構造化マスタ** | スタッフ・ポジション・スキル・役職 | CSVアップロード |
| **B: 半構造化データ** | 出勤希望(原則出勤希望ベース、30分ブロック) | CSVアップロード |
| **C: 非構造化データ** | 個別事情・例外・人間関係 | 自然言語(Streamlitテキスト入力) |

## 設計判断

### ハイブリッド構成を採用

「構造化できるものはCSV、できないものは自然言語」という現実的な分担。

**理由**:
- 30人 × 1ヶ月のリアル規模では完全自然言語化は非効率
- CSVで土台、自然言語で例外対応というワークフローが現場の頭の中と一致
- 自然言語の見せ場が「本当に必要なところ」に集中 → AIエージェントの必然性が立つ
- 未知type検出のシナリオが自然(構造化できない制約 = 自然言語で来る = 未知type候補)

### As-Is / To-Be の整理

| 状態 | 構成 |
|---|---|
| **As-Is(今回作る)** | CSV → DB → エージェント |
| **To-Be(構想)** | スタッフ用フロントエンド → DB → エージェント |

ハッカソン版ではスタッフ入力UIは省略するが、「将来はスタッフ用UI経由で自動化できる」ストーリーを示せる。

### 出勤希望ベース(原則出勤希望)

**変更点**: 引き継ぎ資料は「希望休」ベースだったが、**「原則出勤希望」ベース** に変更。

**理由**:
- 飲食店バイト文化に適合(原則は休み、出勤可日を指定)

**実装上の影響**:
- 引き継ぎ資料 §4 の `availability` type と意味が一致
- CSVに記載のない日時 → **原則出勤不可** として扱う

### 30分ブロック単位

引き継ぎ資料の `slot_minutes: 30|60` の **30分側を採用**。

**影響**:
- 変数空間が約4倍に増える(30人×30日×24スロット = 21,600 セル)
- ただし「シフト開始時刻と終了時刻を変数化」する方式なら変数激減可能
- 計算時間に注意、8秒フェイルセーフが効く可能性あり

## 入力フロー全体図

```
┌─────────────────────────────────────────────────────┐
│ 【初回セットアップ】                                  │
│ ① staff.csv アップロード                            │
│ ② positions.csv, skills.csv, roles.csv アップロード │
└─────────────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────────────┐
│ 【月次シフト作成】                                    │
│ ③ 営業情報フォーム入力                              │
│ ④ desired_shifts.csv アップロード                   │
│ ⑤ 追加制約(自然言語)入力 → 解析                    │
│   ├─ 既知typeに分類 → JSON化                        │
│   └─ 未知type検出 → ハンドラ生成 → テスト → 承認    │
│ [シフト生成]                                        │
└─────────────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────────────┐
│ 【ラリー】                                          │
│ ⑥ フィードバック(自然言語)入力                     │
│ [再計算 (locked 戦略)]                             │
│ → 差分表示                                          │
│ ⑥に戻る(N回繰り返し可)                              │
└─────────────────────────────────────────────────────┘
                    ↓
              [シフト確定]
```

### desired_shifts.csv(出勤希望)

```csv
person_id,date,slot_start,slot_end,priority
p001,2026-12-01,10:00,15:00,must
p001,2026-12-02,17:00,22:00,prefer
p001,2026-12-05,10:00,22:00,available
p002,2026-12-01,17:00,22:00,must
```

| カラム | 用途 |
|---|---|
| person_id | スタッフID |
| date | 対象日 |
| slot_start | 出勤可能開始時刻(30分単位) |
| slot_end | 出勤可能終了時刻(30分単位) |
| priority | must(絶対入りたい) / prefer(できれば) / available(どっちでも) |

## 営業情報フォーム(Streamlit)

CSVではなくフォーム入力。月によって変わる、項目少ない。

```
[期間設定]
  開始日: [📅 2026/12/01]
  終了日: [📅 2026/12/31]
  最小スロット: [🔘30分 / ⚪60分]

[営業時間]
  開店: [⏰ 10:00]
  閉店: [⏰ 23:00]

[基本的な人員配置]  ← マトリックス形式
              月 火 水 木 金 土 日
  ランチ(11-15) 4  4  4  4  5  6  6
  ディナー(17-22) 5  5  5  5  7  8  8

[特別営業日(追加で指定)]
  + 12/24: ディナー +2名
  + 12/25: 終日 +3名
  + 12/31: 22時閉店
  [+ 追加]

[ポリシーモード]
  ○ 希望優先(wishes)
  ● バランス(balance)
  ○ コスト優先(cost)
```

内部的に `frame` JSON + `headcount_requirement` の配列に変換。

## 自然言語入力UI(⑤と⑥)

`text_area` で大きめの入力欄。

```
[追加の特殊事情・制約事項]

以下のような事情があれば自由に記述してください:
- 個別の事情(保育園、介護、副業など)
- 人間関係(同じシフトにしない、ペアにしたいなど)
- 特別な業務要件

┌────────────────────────────────────────┐
│ 田中花子さんは12月から保育園のお迎えがあり、    │
│ 18時までに退勤できるようにしてください。       │
│                                        │
│ 新人の山田美咲さんは1人で勤務させないでください。│
│ 必ずシニア以上を同シフトに配置してください。   │
│                                        │
│ 鈴木一郎さんと佐藤太郎さんは過去に          │
│ トラブルがあったため、できれば            │
│ 同じシフトにしないでください。            │
└────────────────────────────────────────┘

[制約を解析]
```

⑤(初回)と⑥(フィードバック)は同じUI。内部で「初回」「修正」フラグだけ持つ。

## 再計算ラリーの設計

### 再計算パターン

| パターン | トリガー | 動作 |
|---|---|---|
| A. 完全再計算 | ユーザーが「やり直し」 | 全部初期化 |
| B. 部分固定再計算 | 「11月後半だけ調整」 | 前半locked、後半再生成 |
| C. 制約追加再計算 | 新しい制約が来た | 既存解を維持しつつ追加制約を取り込む |
| D. 修正提案再計算 | ユーザーが具体的修正指示 | その部分だけ再計算 |

**デモでは C と D が主役。**

### locked戦略

「前回の解をどこまで尊重するか」の方針:

| 戦略 | 内容 | 採用 |
|---|---|---|
| 戦略1: 全部 locked | 前回解を維持、衝突部分だけ調整 | **ベース** |
| 戦略2: 全部リセット | 毎回ゼロから | 不採用 |
| 戦略3: ユーザー指定 locked | 「ここは絶対動かすな」を明示 | **デモでチラ見せ** |

### 差分表示

再計算後に差分を可視化:

```
[結果差分表示]
  11/3 ディナー: +1名追加(山田)
  11/5: 田中→11/6に移動
  その他: 変更なし
```

エージェントの判断が透明に見えてデモで効く。

## デモシナリオへの貢献(予告)

このラリー構造で組めるデモ:

1. CSV投入(早送り) + 簡単な自然言語制約 → 初回シフト生成
2. 「人手不足な日がある」フィードバック → 追加制約で再計算 → 改善
3. 「保育園のお迎えで」← **未知type!** → ハンドラ生成 → テスト → 承認
4. 承認後、新typeを取り込んで再計算 → 最終シフト完成
5. アーキ図と「スタッフUI連動による完全自動化」のTo-Beビジョン
