**詰めること**:

- 30分ブロック・出勤希望ベースとの整合性
- ハッカソン版で省略すべき項目の特定
- 追加が必要な項目の特定

**ステータス**: ✅ 完了(2026-05-28)
- スキーマは高整合性。矛盾箇所なし
- store_id/user_id/RLS/24h削除 → ハッカソン版では不要と確定
- desired_workdays・break_rule・HH:MM時間表現は完全互換
- 未知3type(recurring_day_off/max_late_shift_count/exam_period)は外部サービスに存在しないことを確認→L2デモ対象として確定

---

### 2. 未知type選び(デモの主役決め)

**ステータス**: ✅ 決定済み(2026-05-27)

**採用する未知type(3つ)**:

| type名 | 入力例 | 既存typeとの違い |
|---|---|---|
| `recurring_day_off` | 「毎週水曜は習い事があって入れません」 | `availability`は日付+時刻の個別指定。こちらは**曜日パターン**で対象期間の全日付に展開する構造 |
| `max_late_shift_count` | 「22時以降まで働くシフトは月3回までにして」 | 既存typeにない**集計型カウント制約**。シフト個数を条件付きで数えて上限を設ける |
| `exam_period` | 「12/10〜20が試験期間なので極力入れないで」 | `desired_workdays`は月全体の希望。こちらは**特定日付範囲だけ**出勤最小化するsoft制約 |

**デモの流れ(3本立て)**:
1. 「毎週水曜は習い事」入力 → `recurring_day_off` 未知検出 → L2フロー → 承認 → 登録
2. 「22時以降は月3回まで」入力 → `max_late_shift_count` 未知検出 → L2フロー → 承認 → 登録
3. 「試験期間は極力入れないで」入力 → `exam_period` 未知検出 → L2フロー → 承認 → 登録
