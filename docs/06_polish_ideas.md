## アイデア1: リクエスト送信前の「要件具体化フェーズ」（AIとの対話で意図を明確化）

*記録: 2026-06-06（テスト中の気づき）*

### 背景・観察
「新人をキッチンに**優先的に**割り当てる」のような**定性的・曖昧な要望**は、解釈が複数あり得る。
AIは1つの解釈を選んでハンドラを生成し、懸念点として申し送るが、**依頼者の本当の意図は汲み切れない**ことが多い。

実例（「新人はキッチン優先」を生成させたときのAIの申し送り）:
- 「優先的に」の解釈として「新人が出勤する日には、少なくとも1コマはキッチンに入る」というロジックを実装した。
  「総労働時間に対するキッチン業務の割合」など、より複雑な意図には対応できない。
- この制約は割り当てに関するもので、新人の出勤日数や時間を増減させる効果はない。
- パラメータの新人IDリスト・キッチンポジションIDリストが、実登録IDと一致している必要がある。

→ つまり「優先」の**定義**（毎日1コマ／割合／他ポジションより多く…）が曖昧なまま実装されている。

### アイデア
ハンドラ生成（L2）に進む**前に、AIが依頼者へ確認質問を投げて要件を具体化する対話フェーズ**を挟む。

```
依頼者: 「新人はキッチン優先で」
   ↓
AI: 「『優先』の意味を確認させてください。次のどれに近いですか？
      (a) 新人が出勤する日は必ず1コマ以上キッチン
      (b) 新人の労働時間の◯割以上をキッチンに
      (c) 他ポジションより新人をキッチンに多く割り当てる」
   ↓
依頼者が選択／補足 → 意図が確定 → その定義でハンドラ生成
```

### 期待効果
- 生成ハンドラの**意図ズレを減らす**（承認の手戻りが減る）
- デモで「AIが曖昧さに気づいて聞き返す」様子は**見せ場になりうる**（賢さの演出）

### 実装の方向性（メモ）
- パーサ段階で「曖昧フラグ」を立て、`review_points` に確認質問候補を持たせる
- ②画面に「AIに確認される→答える」ミニ対話UIを足す（1〜2往復）
- 確定した定義を `suggested_schema` / handler生成プロンプトに渡す
- 自己評価スコア（参考）: 8/10（デモのwow＋実用性。ただしコア完成後でよい）
