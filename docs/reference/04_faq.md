# FAQ：これまでの議論で出た問いと回答

会話の中で出てきた質問を、カテゴリ別に整理。「これってどういう意味？」となったらまずここを見る。

-----

## 📦 カテゴリ1：JSON設計について

### Q1. なぜJSONを使うのか？マスタテーブルだけで良くないか？

**A.** マスタデータ（occupation = 会社員、社長…）自体は**テーブルで管理するのが正しい**。JSONを使うのは「制約の**構造そのものが型ごとに違う**」状況を扱うため。

|案              |問題                                            |
|---------------|----------------------------------------------|
|全カラムを1テーブルに    |NULLだらけ、型増えるたびにALTER TABLE                    |
|型ごとに別テーブル      |クエリでUNIONだらけ、型追加でCREATE TABLE                 |
|EAVテーブル        |1制約が複数行に分割、クエリ複雑化                             |
|**JSON（JSONB）**|**型ごとに違う構造を1カラムで扱え、配列・ネストにも強く、Geminiの出力とも相性◎**|

→ JSONには「マスタへの参照」を入れ、マスタ本体は別途テーブルで管理する。

-----

### Q2. typeフィールドは何の役割？

**A.** **タグ付きユニオン（discriminated union）** のタグ。
「どんなスキーマがそのレコードに入っているか」を示す目印になる。

```json
{
  "type": "separate",         ← この値を見れば
  "targets": ["A", "B"],      ← どのスキーマか分かる
  "priority": "hard"
}
```

プログラムは**まずtypeを読む** → どのスキーマで解釈すべきか判断 → 対応するハンドラを呼ぶ、という流れ。

-----

### Q3. 同じtype内でスキーマは固定？それとも違っても良い？

**A.** **同じtypeなら同じスキーマが鉄則**。

```json
// ✅ OK: 同じtype、値だけ違う
{ "type": "required_on_day", "person": "C", "day": "saturday" }
{ "type": "required_on_day", "person": "D", "day": "sunday" }

// ❌ NG: 同じtypeなのにスキーマが違う
{ "type": "required_on_day", "person": "C", "day": "saturday" }
{ "type": "required_on_day", "person": "D", "shift": "morning" }
```

スキーマが変わるなら**新しいtypeとして扱う**。これを許すとハンドラが壊れる。

-----

## 🔍 カテゴリ2：未知のものの扱い

### Q4. 「未知」には2種類あるって本当？

**A.** はい、レイヤーが2つある。

|レイヤー     |何が未知か             |対処                     |
|---------|------------------|-----------------------|
|1：スキーマレベル|新しい`type`が登場      |新type＋ハンドラを生成（L2の自動化対象）|
|2：値レベル   |既知typeだがパラメータ値が新しい|マスタに新値を追加              |

-----

### Q5. 値レベルの未知ってどんなこと？

**A.** 例えば、

```
入力1: 「サラリーマンのAさんは土曜出勤」
入力2: 「会社員のBさんは日曜出勤」
```

人間には「サラリーマン」と「会社員」が同じと分かるが、プログラムには別物。
→ **マスタ＋正規化**で解決する。

```
[会社員] というマスタ項目
  ├─ 別名: サラリーマン
  ├─ 別名: 社員
  └─ 別名: 一般職
```

AIが入力を見るとき、マスタ照合してから正規化された値でJSONに保存。

-----

### Q6. 完全に未知の値が来たら？

**A.** 3つに分岐する。

1. **似た既存項目を提案**：「これは会社員に近いです」
1. **新規マスタ追加を提案**：人間承認待ち
1. **解釈不能フラグ**：人間にエスカレーション

-----

## 🔧 カテゴリ3：ハンドラ関数とソルバー

### Q7. ソルバーって何？

**A.** ルール（制約）と目標を渡すと、それを満たす最適な答えを計算してくれるエンジン。SudokuやNumberlinkを解くプログラムも仲間。

シフト管理での流れ：

1. 変数を定義：`x[A][月] = 1なら出勤、0なら休み`
1. 制約を渡す：`x[A][月] + x[B][月] <= 1`（同日NG）
1. 目標を渡す：`minimize 出勤日数の不均衡`
1. `solver.solve()` で計算

代表的ソルバー：**Google OR-Tools**（無料・定番）、Gurobi（商用）、CPLEX（商用）

-----

### Q8. ハンドラ関数のアウトプット形式は？Pythonの不等式？

**A.** **ほぼPythonの不等式そのもの**。ソルバーAPIを呼ぶコード。

```python
def handle_separate(constraint, model, x, days):
    target1 = constraint["targets"][0]
    target2 = constraint["targets"][1]
    for day in days:
        model.Add(x[target1][day] + x[target2][day] <= 1)
```

ソルバーが理解できるのは「変数」と「変数同士の不等式・等式」だけ。

- ✅ `x <= 5`, `x + y == 1`, `sum(shifts) >= 2`
- ❌ “離れて配置”、“なるべく公平に”、“雰囲気を悪くしない”

-----

### Q9. ハンドラ関数は毎回AIが生成するの？

**A.** **いいえ、1回だけ**。新typeが初登場したとき1回生成→辞書に永続登録→以降は同じものを使い回す。

|観点 |理由                      |
|---|------------------------|
|速度 |ソルバー実行時にAI呼び出しを挟むと遅い    |
|安定性|コード生成は毎回微妙に揺らぐ、本番では固定が安全|
|コスト|毎回Gemini APIを叩くと費用がかさむ  |
|テスト|「1度書いたら同じ動き」の方がデバッグしやすい |

AIは**「新しい翻訳辞書を作る人」**、運用時はその辞書を使い回す。

-----

### Q10. ソルバーが受け取る情報の粒度は？

**A.** 「変数」と「変数同士の不等式・等式」のみ。

|ソルバーが扱える                     |ソルバーが扱えない|
|-----------------------------|---------|
|`x <= 5`                     |“離れて配置”  |
|`x + y == 1`                 |“保育園のお迎え”|
|`sum(x for x in shifts) >= 2`|“なるべく公平に”|

意味の理解はAIの仕事、計算はソルバーの仕事、と役割分担されている。

## ⚠️ カテゴリ4：自動化の限界と対処

### Q11. ハンドラ関数が無理やり翻訳して精度落ちない？

**A.** **起こり得る**。これがこの種のシステム最大の弱点。Tier別に対処する。

|Tier   |翻訳精度|対処         |
|-------|----|-----------|
|1: きれい |完璧  |自動承認OK     |
|2: 分解可能|良好  |テスト通過で自動承認 |
|3: 無理あり|危ない |自信度低→人間レビュー|
|4: 翻訳不能|不能  |エスカレーション   |

精度劣化の主原因：

1. **ソフトvsハードの取り違え**（「離してほしい」を絶対禁止に翻訳）
1. **量化詞の喪失**（「なるべく」が消える）
1. **ドメイン知識の欠落**（「ベテラン」「繁忙期」が不明）

-----

### Q12. Gemini Flashで品質大丈夫？

**A.** **シンプルな制約なら十分。複雑なものは怪しい。**

|制約例              |Flash対応度  |
|-----------------|----------|
|AとBを離す           |✅ 余裕      |
|連続勤務3日まで         |✅ 問題なし    |
|新人には必ずベテランを隣に    |⚠️ 関係性の定義次第|
|繁忙期は売上予測に基づいて人員調整|❌ 厳しい     |

対処：自信度ベースのルーティング（自動マージ / 人間レビュー / 解釈不能フラグ）。
**完全自動化より「信頼できる自動化」を目指す**。

-----

### Q13. 完全自動化と半自動化、どちらが良い？

**A.** **半自動化（AI提案＋人間承認）が圧倒的に良い**。

完全自動化が破綻する理由：

- LLMが勝手にスキーマを生やすと、計算エンジン側のコードが意味を解釈できない事故が止められない
- 評価の自動化が困難（新制約が正しく機能しているかの正解がない）

ハッカソン的にも、「AIが解けない領域を自覚して人間に渡す」設計の方が**エージェントとしての価値が高く評価される**。

## 📊 カテゴリ5：L1とL2

### Q14. L1とL2の違いは何？

**A.** 「マスタに名前を増やすだけ」がL1、「処理コードまで自動生成」がL2。

|       |L1のみ       |L2まで    |
|-------|-----------|--------|
|新type検出|✅          |✅       |
|ハンドラ生成 |❌（人間が書く）   |✅（AIが書く）|
|結果     |開発者対応まで動かない|承認後すぐ動く |

L1だけだと辞書に名前が増えるだけで、プログラムはまだ新制約を解けない。
L2まで行って初めて**自律的なエージェント**になる。

-----

### Q15. L1だけだとソルバー本体の改修が必要？

**A.** **ソルバー本体は触らない。「ハンドラ関数」が必要。**

```
[新制約JSON]
    ↓
[ハンドラ関数] ← ここを追加する必要がある
    ↓ ソルバーAPIを呼ぶ
[ソルバー本体（OR-Toolsなど）] ← 触らない
```

ソルバー本体は汎用品で、不等式しか理解しない。ハンドラ関数の仕事は「JSONを読んで、ソルバーAPIを正しく呼ぶ」だけ。L2はこのハンドラを自動生成する。

## 🔌 カテゴリ6：外部サービスとの連携

### Q16. 独立開発した方が良い？組み込みは大変？

**A.** **独立開発がおすすめ。組み込みは半日仕事で済む**。

設計の2パターン：

|パターン             |結合度|デモ             |移植性|
|-----------------|---|---------------|---|
|A: 疎結合（仕様書を返すAPI）|弱  |独立             |高  |
|B: 密結合（直接ソルバー叩く） |強  |外部サービス前提|低  |

→ **パターンA採用**。L2は「制約仕様書」を返すAPIに徹し、利用側で自分のソルバーに翻訳。

```
POST /parse-constraints
入力: { "text": "..." }
出力: { type, params, is_new_type, needs_review }
```

組み込みはこれを叩く関数1個追加するだけ。

## 💰 カテゴリ7：最適化・継続改善

### Q17. ハンドラ生成だけ高品質モデル、JSON変換は安価モデルでいい？

**A.** **その通り。これはモデルカスケードと呼ばれる定番パターン**。

|用途         |推奨モデル       |頻度        |
|-----------|------------|----------|
|自然言語→JSON変換|Gemini Flash|毎回（多い）    |
|ハンドラコード生成  |Gemini Pro  |新type登場時のみ|

ハンドラ生成は新typeが登場したときだけなので、高品質モデルを使ってもコストはそれほど跳ねない。全体コストの大半は安価なFlashが占める構造になる。

**さらに+α**：LLM as a Judge パターン

- Flashで翻訳 → Proで「この翻訳は正しい？」と判定
- OK: 採用 / NG: Proが書き直し
- 普段はFlashだけ、危なそうなときだけProが介入する設計

-----

### Q18. ユーザー評価をモデル学習に使うのはアリ？

**A.** **アリだし、ハッカソンの「まわす」テーマに完璧にハマる**。ただし設計上の注意点がある。

#### 注意点①：帰属問題（Attribution Problem）

総合点1つだと「どこに問題があったか」が特定できない。
→ 観点別フィードバック（公平性／希望反映／繁忙期対応など）にする。

#### 注意点②：何を学習するか

|学習対象       |難易度 |効果      |
|-----------|----|--------|
|ソフト制約の重み調整 |★★☆ |中（実装現実的）|
|ハンドラ品質ランキング|★★☆ |中       |
|制約間矛盾の検出   |★★★ |高       |
|完全な再生成判定   |★★★★|高       |

→ ハッカソンでは「ソフト制約の重み調整」が現実的。線形回帰やベイズ最適化で実装可能。

#### 注意点③：コールドスタート問題

学習データゼロから始められない。
→ 初期は合成データで起動 → 実データで上書きしていく。

-----

### Q19. このフィードバックループ、何が嬉しいの？

**A.** 「**自走するDevOpsサイクル**」を実現できる。

```
[シフト出力]
    ↓
[ユーザー評価（次元別）]
    ↓
[学習・改善]
  - ソフト制約の重みを自動調整
  - 怪しいハンドラをフラグ立て
    ↓
[次回のシフト品質UP]
```

これは単発のAI機能ではなく、**システムが継続的に賢くなる**設計。
審査ポイントの「自律性」「DevOps」をどちらも満たせる。
